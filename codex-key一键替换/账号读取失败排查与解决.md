# Error: account/read failed during TUI bootstrap 排查与解决

## 现象
- 执行 `codex` 启动 TUI 时提示：
  ```
  Error: account/read failed during TUI bootstrap
  ```
- 常见于 **升级 codex-cli（如 0.117.0）之后**。
- 使用自建/代理服务（如 CRS）时更容易出现。

## 原因
- **codex-cli 0.117.0 在 TUI 启动阶段会读取账号信息（`account/read`）**。
- 当前 `base_url` 指向的 CRS 服务 **未实现 `account/read` 接口**，导致返回 404。
- 配置中 `requires_openai_auth = true` 会触发该账号读取步骤，从而启动失败。

## 解决方法
> 核心思路：要么让服务端支持 `account/read`，要么关闭该校验。

### 方案 A：关闭账号读取校验（推荐，最快恢复）
1. 备份配置与登录态：
   ```bash
   cp ~/.codex/config.toml ~/.codex/config.toml.bak-$(date +%F-%H%M)
   mv ~/.codex/auth.json ~/.codex/auth.json.bak-$(date +%F-%H%M)
   ```
2. 修改配置：
   ```bash
   sed -i 's/requires_openai_auth = true/requires_openai_auth = false/' ~/.codex/config.toml
   ```
3. 重新运行：
   ```bash
   codex
   ```

### 方案 B：关闭 app-server TUI
- 临时（仅这次启动）：`codex --disable tui_app_server`
- 永久（写到配置）：在 `C:\Users\<用户名>\.codex\config.toml` 加上

```
[features]
tui_app_server = false
```

### 方案 C：让服务端支持 `account/read`
- 在 CRS 服务中新增 `GET /account/read`（或与 codex-cli 期望路径一致）。
- 能正确返回账号信息后，保持 `requires_openai_auth = true` 即可。

### 方案 D：回退 codex-cli 版本（临时手段）
- 若短期无法改服务端，可回退到升级前能正常使用的版本。

## 验证方法
```bash
# 查看当前 base_url
rg -n "^\s*base_url\s*=\s*\"" ~/.codex/config.toml

# 验证 account/read 是否可访问（注意替换 token / 路径）
curl -i -H "Authorization: Bearer $CRS_OAI_KEY" http://<host>:<port>/openai/account/read
```

## 备注
- 如果仍报错，请确认：
  - `base_url` 是否正确
  - 服务端是否支持 `responses` / `account/read`
  - 环境变量 `CRS_OAI_KEY` 是否有效
